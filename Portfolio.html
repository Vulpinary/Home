Создание и тестирование API с использованием FastAPI: Практический опыт

В процессе поиска возможностей для применения и оттачивания навыков, возникла потребность в проекте, который позволил бы работать с API, взаимодействующим с базой данных. К сожалению, подходящих API в открытом доступе найти не удалось.

Тогда, несмотря на отсутствие опыта в программировании, я решила попробовать создать свой собственный проект с нуля. Так родилась идея разработки API, привязанного к базе данных. Этот вызов позволил мне взглянуть на процесс разработки с точки зрения разработчика и тестировщика одновременно.

Первым шагом стало визуальное представление структуры проекта в виде майнд-карты. Затем был создан набросок требований к API (который, безусловно, требует дальнейшей доработки). На основе этих требований были разработаны чек-лист и тест-кейсы.

После этого я приступила к тестированию API. Начала с ручного тестирования, чтобы понять логику работы приложения и выявить основные дефекты. Важно отметить, что, поскольку этот проект носит учебный характер, исправление всех найденных багов не являлось приоритетной задачей. После выявления уязвимых мест и особенностей работы API, я приступила к подготовке окружения и настройке проекта автотестов на языке Python, чтобы продемонстрировать свой уровень знаний и практические навыки в автоматизации API-тестирования.

В результате был создан проект, включающий следующие компоненты:

  • FastAPI (Мой тестовый API): Разработанный API, служащий объектом тестирования.
  • Требования к продукту: Первичный набросок целей и функциональности API (требует доработки).
  • Майнд-карта FastAPI: Визуальное представление структуры и взаимосвязей компонентов API.
  • Чек-лист: Список проверок для обеспечения качества API.
  • Тест-кейсы: Сценарии тестирования для ручной проверки функций API.
  • Баг-репорты: Описание выявленных в процессе тестирования ошибок и проблем.
  • Отчет о тестировании и рекомендации: Результаты тестирования, анализ выявленных проблем и рекомендации по улучшению.
  • Автотесты: Автоматизированные тесты для проверки API (в разработке).

Этот проект наглядно демонстрирует мои навыки в тестировании API, а также стремление к пониманию процесса разработки ПО и постоянному совершенствованию профессиональных навыков.

FastAPI (Мой тестовый API)

Созданный мной тестовый API, использованный для сбора портфолио. По ссылке https://vulpinary.github.io/Vulpy/#/ находится демонстрация фрагментов этого API, но запросы не отправляются, так как GitHub передает только статические данные.

API включает в себя регистрацию и авторизацию продавцов, создание категорий продукции и другие функции.

Требования к продукту

  • 1. Регистрация продавца

    Функциональность: Продавец должен иметь возможность зарегистрироваться в системе.

    • username: уникальное имя пользователя (строка, максимум 50 символов).
    • password: пароль (строка, минимум 8 символов, должен содержать нижний регистр, верхний регистр и цифры).
    • name: полное имя (строка, максимум 100 символов).
    • email: адрес электронной почты (строка, уникальная, максимум 100 символов).

    Выходные данные: Подтверждение успешной регистрации или сообщение об ошибке «имя пользователя или адрес электронной почты уже заняты».

  • 2. Авторизация продавца

    Функциональность: Продавец должен иметь возможность войти в систему.

    • username: имя пользователя.
    • password: пароль.

    Выходные данные: доступ к токену для последовательных запросов (с жизнью токена 60 минут) или сообщение об ошибке «неверное имя пользователя или пароль».

  • 3. Создание категории продукции

    Функциональность: Продавец должен иметь возможность создавать категории для продукции.

    • name: название категории (строка, уникальный, максимум 255 символов).

    Выходные данные: Подтверждение получения категории или сообщение об ошибке «категория с таким именем уже существует».

  • 4. Создание поставщика

    Функциональность: Продавец должен иметь возможность создавать поставщиков и присваивать категорию.

    • name: название поставщика (строка, максимум 55 символов).
    • email: адрес электронной почты (опционально).
    • phone: номер телефона (строка, максимум 11 символов).
    • address: адрес поставщика (строка, максимум 50 символов).
    • category_id: ID категории, к которой относится поставщик (целое число).

    Выходные данные: Подтверждение получения поставщика.

  • 5. Создание продукта

    Функциональность: Продавец должен иметь возможность создавать продукты и присваивать категорию и поставщика каждого продукта.

    • name: название продукта (строка, максимум 25 символов).
    • manufacturer: производитель продукта (строка, максимум 25 символов).
    • price: цена товара (х.хх).
    • description: описание товара (опционально).
    • category_id: ID категории продукта (целое число).
    • supplier_id: ID поставщика продукта (целое число).

    Выходные данные: Подтверждение получения продукта или сообщение об ошибке.

  • 6. Получение списка продуктов

    Функциональность: Продавец должен иметь возможность получать все свои продукты.

    Выходные данные: Список объектов продуктов с их данными.

  • 7. Получение продукта по ID

    Функциональность: Продавец должен иметь возможность получать информацию об конкретном продукте по его идентификатору.

    • product_id: ID продукта (целое число).

    Выходные данные: Объект продукта с его данными или сообщение об ошибке, если продукт не найден.

  • 8. Получение категорий

    Функциональность: Продавец должен иметь возможность получать товары для всех своих категорий.

    Выходные данные: Список категорий с их данными.

  • 9. Получение категории по ID

    Функциональность: Продавец должен иметь возможность получать информацию о конкретной категории по ее идентификатору.

    • category_id: ID категории (целое число).

    Выходные данные: Категория объекта с ее данными или сообщение об ошибке, если категория не найдена.

  • 10. Получение списка поставщиков

    Функциональность: Продавец должен иметь возможность получать список всех своих поставщиков.

    Выходные данные: Список объектов поставщиков с их данными.

  • 11. Получение идентификатора поставщика

    Функциональность: Продавец должен иметь возможность получать информацию о конкретном поставщике по его идентификатору.

    • supplier_id: ID поставщика (целое число).

    Выходные данные: Объект поставщика с его данными или сообщение об ошибке, если поставщик не найден.

  • 12. Обновление продукта

    Функциональность: Продавец должен иметь возможность обновлять информацию о продукте.

    • product_id: ID продукта (целое число).

    По получения обновления продукта или сообщение об ошибке.

  • 13. Обновление поставщика

    Функциональность: Продавец должен иметь возможность обновлять информацию о поставщике.

    • supplier_id: ID поставщика (целое число).

    Поля для обновлений:

    • name
    • email
    • phone
    • address
    • category_id

    Выходные данные: Подтверждение получения обновления поставщика или сообщение об ошибке.

  • 14. Обновление категории

    Функциональность: Продавец должен иметь возможность обновлять информацию о категории.

    • category_id: ID категории (целое число).

    Поля для обновлений:

    • name

    Выходные данные: Подтверждение получения обновления категории или сообщение об ошибке.

  • 15. Обновление продавца

    Функциональность: Продавец должен иметь возможность обновлять свою информацию.

    Поля для обновлений:

    • Name
    • email

    Выходные данные: Подтверждение получения обновления информации о продавце или сообщение о необходимости.

  • 16. Удаление продукта

    Функциональность: Продавец должен иметь возможность удалить продукт по идентификатору.

    • product_id: ID продукта (целое число).

    Выходные данные: Подтверждение удаления продукта или сообщение об ошибке.

  • 17. Удаление поставщика

    Функциональность: Продавец должен иметь возможность удалить поставщика по идентификатору.

    • supplier_id: ID поставщика (целое число).

    Выходные данные: Подтверждение удаления поставщика или сообщение об ошибке.

  • 18. Удаление категории

    Функциональность: Продавец должен иметь возможность удалить категорию по идентификатору.

    • category_id: ID категории (целое число).

    Выходные данные: Подтверждение удаления категории или сообщение об ошибке.

  • 19. Удаление продавца

    Функциональность: Продавец должен иметь возможность удалить свой идентификатор.

    • seller_id: ID продавца (целое число).

    Выходные данные: Подтверждение удаления продавца или сообщение об ошибке.

  • Общие требования

    • 1) Связь с продавцом

      Все создаваемые объекты (продукты, категории, поставщики) должны быть связаны с продавцом через поле seller_id. Это гарантирует, что каждый продавец может управлять только собственными данными.

    • 2) Интеграция Swagger с базой данных

      Пользовательский интерфейс Swagger должен быть интегрирован в приложение для удобного API тестирования. Все манипуляции с эндпоинтами должны осуществляться на основе данных, которые позволяют пользователям видеть изменения в первый момент и проверять корректность работы API.

Отчет о тестировании и рекомендации

Введение

В рамках тестирования API были выявлены критические уязвимости и недостатки в работе системы, которые необходимо устранить для повышения безопасности, стабильности и соответствия спецификациям API.

Выявленные баги

  • (API) Создание продукта с пустым именем: POST-запрос к `/api/products` позволяет создать продукт с пустым именем, что нарушает валидацию данных. Ожидается ошибка 400 Bad Request.
  • (API) Обновление данных продукта не обновляет данные: PUT-запрос запрос к `/api/products/{id}` не сохраняет изменения данных продукта. Ожидается код 200 OK с обновленными данными.
  • (API) Изменение поля Username: PUT-запрос к `/api/sellers/{id}` возвращает код 200 OK, но имя пользователя фактически не меняется. Требуется уточнение логики и кодов ответа.
  • (API) Регистрация с невалидным паролем: POST-запрос к `/api/register` позволяет зарегистрировать пользователя с паролем, не соответствующим требованиям безопасности. Ожидается ошибка 400 Bad Request.
  • (API) Создание поставщика с пустым именем: POST-запрос к `/api/suppliers` позволяет создать поставщика с пустым именем. Ожидается ошибка 400 Bad Request.

Улучшения продукта

  • (API) Валидация данных: Внедрить строгую валидацию данных на уровне API для всех эндпоинтов, создающих или обновляющих данные (продукты, поставщики, пользователи). Обязательно проверять наличие и формат обязательных полей.
  • (API) Безопасность пароля: Реализовать проверку пароля на соответствие требованиям сложности (минимальная длина, наличие символов верхнего и нижнего регистра, цифр и специальных символов) при регистрации и смене пароля. Возвращать ошибки валидации при несоответствии.
  • (API) Управление пользователями:
    • Уточнить логику эндпоинта для изменения имени пользователя. Если изменение запрещено, возвращать код 403 Forbidden.
  • (API) Коды ответа:
    • Исправить коды ответа API в соответствии со спецификацией REST:
      • Возвращать 400 Bad Request для невалидных запросов.
      • Возвращать 404 Not Found для запросов к несуществующим ресурсам.
      • Возвращать 409 Conflict при попытке создания дубликатов (например, пользователя с уже существующим email).

Рекомендации

  • (API) Обработка ошибок: Внедрить централизованную систему обработки ошибок API для обеспечения единообразного и информативного ответа клиентам. Включить журналирование ошибок для упрощения отладки.

Заключение

Устранение выявленных уязвимостей и внедрение предложенных улучшений значительно повысят качество, надежность и безопасность API, что положительно скажется на всей системе.
